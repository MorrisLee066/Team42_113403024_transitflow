# Task 6 Extension - Frequency-Based Departure-Time Booking

## Motivation

The original national rail mock data includes `first_train_time`,
`last_train_time`, and `frequency_min`, but these timetable fields were not
fully preserved and validated throughout the original booking workflow.

National rail services are represented as repeating frequency-based schedules
rather than as one database row for every individual departure. Without
departure-time validation, the booking operation could accept a time that was
not generated by the selected schedule.

This extension improves TransitFlow by:

- preserving the complete service window in PostgreSQL;
- generating valid departure times dynamically;
- adjusting departure times based on the selected origin station;
- returning valid departure times in availability results;
- rejecting missing or invalid departure times before creating a booking;
- ensuring availability and booking use the same timetable-generation logic.

## Files Modified

### `databases/relational/schema.sql`

Database objects modified:

- `national_rail_schedules`

Columns added or preserved for the extension:

- `first_train_time`
- `last_train_time`
- `frequency_min`

These fields store the service window and interval used to generate valid
national rail departure times.

### `skeleton/seed_postgres.py`

Functions or operations modified:

- national rail schedule seed-data preparation;
- `INSERT INTO national_rail_schedules` operation.

The seed process now reads and inserts:

- `first_train_time`
- `last_train_time`
- `frequency_min`

from `train-mock-data/national_rail_schedules.json`.

### `databases/relational/queries.py`

Functions modified or added:

- `_generate_departure_times()`
- `query_national_rail_availability()`
- `execute_booking()`

`_generate_departure_times()` generates the valid departure-time list from:

- `first_train_time`
- `last_train_time`
- `frequency_min`
- `travel_time_from_origin_min`

`query_national_rail_availability()` returns the generated departure-time list
for each matching schedule.

`execute_booking()` regenerates the same valid departure-time list and rejects:

- missing departure times;
- departure times that are not valid for the selected schedule and origin.

### `skeleton/agent.py`

Agent components modified:

- booking tool description;
- `departure_time` tool parameter;
- departure-time forwarding to `execute_booking()`;
- booking workflow instructions.

The agent is instructed to retrieve valid departure times before submitting a
booking request.

## Implementation Summary

Departure times are generated dynamically using:

- `first_train_time`
- `last_train_time`
- `frequency_min`
- `travel_time_from_origin_min`

The selected origin station's `travel_time_from_origin_min` is added to the
schedule-origin service window. This ensures that the departure times shown for
an intermediate station are shifted correctly.

A fallback service window of `06:00` to `23:00` is retained only as defensive
behaviour for incomplete legacy data. Under normal operation, the values are
read from the seeded database record.

The availability query and booking operation share the same
`_generate_departure_times()` function. This prevents the system from showing
one set of departure times while accepting a different set during booking.

`execute_booking()` rejects a request when:

- `departure_time` is missing;
- the requested time is not present in the generated valid departure-time list.

When validation fails, the transaction is rolled back and no booking or payment
record is created.

The existing seat-conflict check is also preserved. It checks:

- schedule;
- travel date;
- departure time;
- seat code;
- active booking status;
- overlapping journey segments.

Booking and payment records are inserted within the same atomic transaction.

The booking operation also uses row-level locking and a PostgreSQL transaction
advisory lock to reduce race-condition risks during concurrent seat selection.

## Testing Evidence

The extension was tested using direct database queries and Python function
calls.

The completed tests verify that:

- schedule service-window fields are stored in PostgreSQL;
- generated `departure_times` is non-empty;
- departure times include the selected origin station offset;
- a missing departure time is rejected;
- an invalid departure time is rejected;
- a valid departure time creates one booking and one linked payment;
- a duplicate overlapping booking for the same seat and departure is rejected;
- the rejected booking leaves no additional booking or payment records;
- the existing cancellation and refund workflow still works after the
  extension.
